# 三、视角对照：地球上已有的「准月球基地」复杂系统

很多"地球上的超级工程"，本质上已经非常接近"月球基地"的复杂系统形态了——只是它们还没有脱离地球环境。

- **国际空间站（ISS）**：长期封闭、资源受限、强耦合的人类生存系统。生保、能源、热控、姿控、通信、补给、乘员心理、任务调度全部互相影响
- **现代数字化核电站**：冗余 / 安全联锁 / 故障传播 / 状态监测 / 人机协同——已接近"复杂自治系统"
- **大型商业飞机（Boeing 787 / Airbus A350）**：航电、液压、电源、飞控、环境控制、传感器、软件高度耦合——787 被称为"会飞的数据中心"
- **特斯拉超级工厂 / 半导体晶圆厂**：能源、水循环、物流机器人、MES、供应链、环境控制、质量追踪实时协同；晶圆厂尤其对稳定性与流程耦合要求极高
- **数据中心 / 太空算力中心**：未来的"数字生命保障系统"。供电、液冷、网络、调度、容灾、AI 推理资源动态耦合
- **现代航天发射场（SpaceX Starbase 等）**：推进剂、测试、发射窗口、气象、供应链、遥测、地面设备、飞控、人机协同——已经非常接近"地外基地"的组织形态

> Harnetics 切中的不是单个 AI 功能，而是这些系统最难的一层：**复杂系统之间如何长期保持一致性与可演化性**。复杂系统真正贵的地方不是制造硬件，而是跨部门协同、接口对齐、变更传播、状态同步、文档一致性、人与系统之间的信息损耗。
> 

> 很多人会把"未来月球基地"理解成建筑问题；但实际上它更像一个**长期运行的 AI-Native Systems Engineering 问题**。

# 四、现存复杂系统工程的主流解决方案

抽象来看，复杂系统工程本质上在解决四件事：

1. 系统如何被"建模"
2. 系统如何"协同"
3. 系统如何"感知状态"
4. 系统如何"应对变化"

## 1. MBSE（模型驱动系统工程）

航天、军工、核工业里最核心的方法之一。

- 典型代表：IBM DOORS / Cameo Systems Modeler / SysML / SysML v2 / OPM（Object-Process Methodology）
- 核心思想：**不再让文档成为系统本身，而让模型成为系统本身**。需求、接口、状态机、功能流、风险、验证关系全部进入统一模型；文档只是模型的"投影"

## 2. 数字孪生（Digital Twin）

工业界最火的方向之一。

- 典型代表：Siemens / GE Aerospace / NVIDIA Omniverse
- 核心思想：**让现实系统在数字世界里始终存在一个同步映射**。发动机温度、能源状态、机器人位置、结构疲劳、故障传播实时同步
- 用途：提前预测故障 / 仿真 / 调度 / 运维优化。ISS、飞机发动机、晶圆厂都在大量使用

## 3. SCADA / 工业控制系统

工业世界最经典的一层。

- 核心：**感知 + 控制 + 告警 + 自动化联锁**
- 应用：电网、核电站、化工厂、水处理系统——传感器 → 控制系统 → 执行器 → 反馈回路
- 很多月球基地论文其实就是这个架构的延伸

## 4. PLM（产品生命周期管理）

- 典型代表：Teamcenter / Windchill
- 解决：**一个复杂产品在几十年生命周期内如何保持一致**——BOM / CAD / 版本 / 工艺 / 测试 / 供应链 / 维修记录
- 本质上是在做"工业世界的知识图谱"

## 5. DevOps / GitOps（软件工程世界）

软件行业其实是最早解决"复杂协同"的。

- 核心思想：Everything as Code / 可追溯 / 自动化 / CI-CD / 状态一致性
- 典型代表：Kubernetes / GitHub Actions / Terraform
- 解决"几千台机器如何持续协同运行"——未来复杂工程系统也会逐渐"软件化"

## 6. AI Agent / Autonomous Operations（正在出现）

最前沿的一层。

- 方向：AI 运维 / 自动调度 / 智能故障分析 / Agent 协同 / 自治系统
- 应用：自动驾驶 / 智能电网 / 无人机蜂群 / 太空自主运行
- 最大问题：**不可验证**。而航天、核工业最怕的就是不可预测

<aside>
💡

**共同问题**：现有方案都在"局部智能化"，但缺少真正统一的"系统语义层"——文档、模型、实时状态、人类决策、AI 推理、变更传播还没有被统一。

Harnetics 的有趣之处在于：把复杂系统里的知识、依赖与变化关系，变成一个可被 AI 理解和推演的**系统语义网络**——这非常接近 **AI-Native Systems Engineering**。

</aside>

# 五、72 小时该怎么打：SysML v2 入门 + AI-Native SE 最小闭环

## 心法

> 不要把目标定成"学会 SysML v2"；而是定成「**跑通一个 AI-Native Systems Engineering 的最小闭环（MVP）**」。
> 
- Harnetics 是 **Document-first**，SysML v2 是 **Model-first**——应该做的是「把两者接起来」，而不是推翻重来
- 不要变成"学习 Cameo 怎么画图"——重 GUI、重流程、重建模语法、重工程仪式感会直接把 72 小时烧光
- SysML v2 的本质只有一句：**用统一语义描述复杂系统之间的关系**

## 黑客松目标

<aside>
🛰️

**AI-Native Lunar Base Systems Copilot**——AI 理解月球基地系统之间的关系，并分析变化传播。

</aside>

## 最小 MVP：四层架构

### Layer 1：系统建模（最小 SysML v2）

只学这几个核心概念：

- Requirement
- Part
- Interface
- Dependency
- State

建模 6 个系统即可：Solar Panel / Battery / Water Recycler / Oxygen System / Habitat / Rover。结构示意：

```
[Energy System]
    ↓
[Oxygen Generator]
    ↓
[Habitat]
```

### Layer 2：文档图谱（Harnetics 的优势区）

输入：

```
oxygen_requirement.md
battery_spec.md
habitat_interface.md
```

解析后建立关系：

```
Requirement -> Component
Component  -> Interface
Interface  -> Dependency
```

然后放入 Graph。

### Layer 3：Impact Analysis（核心亮点）

用户输入示例：

> 月夜持续 14 天，太阳能下降 40%
> 

AI 自动分析：

```
Energy ↓
  → Battery Stress ↑
    → Oxygen Production ↓
      → Habitat Risk ↑
```

这就是**复杂系统变化传播**——已经非常像真正的 Systems Engineering。

### Layer 4：AI Copilot（灵魂）

让 Agent 不只是聊天，而是"理解系统结构"。例如：

> 问：如果氧气系统故障会影响什么？
> 

> AI：
> 

> - Habitat pressure stability
> 

> - Water recycle loop
> 

> - Crew survival duration
> 

> - EVA scheduling
> 

这一步会让项目从"AI Demo"变成「**AI-Native Systems Engineering**」。

## 必学（72h 内即可掌握的 SysML v2 子集）

1. **Requirement Definition** — 例：`Habitat shall support 4 crew for 30 days`
2. **Part Definition** — 例：`part def Habitat` / `part def Battery` / `part def OxygenGenerator`
3. **Connection / Interface** — 系统之间怎么连接
4. **Dependency** — 变化传播，**最关键**

## 技术路线（直接复用现有 Harnetics 栈）

- **建模层**：SysML v2 text notation（不用 GUI）
- **存储层**：SQLite + Graph
- **AI 层**：RAG + Graph Retrieval + Agent
- **Demo 层**：CLI 就够。**评委真正想看的是系统思维，不是前端动画**

## 最值得探索的一件事

<aside>
🚀

不要把 Harnetics 定义成"AI + 文档工具"，而是「**AI-Native Systems Engineering Runtime**」。

传统 MBSE 是静态、人工维护、重流程、重建模；
