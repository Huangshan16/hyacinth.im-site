# 我的月球 Demo 数值交互 Spec

## 1. 设计目标

这份 spec 解决一个问题：

`Demo 里的数值不能只是几条独立进度条，而要形成可传播、可解释、可调度的系统关系。`

来源文档给出的最小闭环是：

- `Part`：系统由哪些部件组成
- `Interface`：部件之间如何连接
- `Dependency`：一个参数变化如何影响下游
- `State`：系统当前处于什么模式

对应到“我的月球”，最小传播链收敛为：

`太阳输入下降 -> 能源舱发电下降 -> 电池压力上升 -> 氧气/舱压/温控供能受限 -> 居住舱风险上升 -> AI 调度负载`

## 2. 建模单位

本 Demo 使用 `EU` 作为演示能源单位，不直接伪装成真实航天工程参数。

- `EU/s`：每秒可供或消耗的演示功率
- `%`：储能、风险、安全等级等归一化指标
- `tick`：模拟步长，推荐 `1 tick = 1 秒演示时间`

这样做的好处是数值关系稳定可控，讲解时也不会陷入真实工程单位校准。

## 3. Part 模型

### 3.1 EnergyCabin

能源舱负责生产、储存和分配能源。

| 字段 | 类型 | 初始值 | 说明 |
| --- | --- | ---: | --- |
| `solarInput` | percent | `100` | 当前太阳输入强度 |
| `solarHealth` | percent | `100` | 能源舱发电系统健康度 |
| `generation` | EU/s | `80` | 当前发电功率 |
| `batterySoc` | percent | `72` | 储能剩余比例 |
| `batteryStress` | percent | `0` | 电池压力 |
| `outputLimit` | EU/s | `90` | 能源舱最大对外输出 |
| `reserveFloor` | percent | `20` | 应急储备下限 |

计算：

```text
generation = 80 * solarInput / 100 * solarHealth / 100
```

### 3.2 HabitatCabin

居住舱负责承载生命支持和生活负载。

| 负载 | 正常需求 | 保底需求 | 优先级 | 可调度性 |
| --- | ---: | ---: | ---: | --- |
| `oxygenLoop` | `18 EU/s` | `16 EU/s` | `100` | 不可关闭 |
| `pressureControl` | `14 EU/s` | `12 EU/s` | `95` | 不可关闭 |
| `thermalControl` | `22 EU/s` | `14 EU/s` | `80` | 可降级 |
| `waterRecycler` | `10 EU/s` | `6 EU/s` | `65` | 可降级 |
| `lighting` | `8 EU/s` | `2 EU/s` | `35` | 可降级 |
| `comfortLoad` | `12 EU/s` | `0 EU/s` | `20` | 可关闭 |
| `publicDisplay` | `6 EU/s` | `0 EU/s` | `10` | 可关闭 |

核心需求：

```text
normalDemand = 90 EU/s
criticalDemand = 48 EU/s
```

其中：

```text
criticalDemand = oxygenLoop.min + pressureControl.min + thermalControl.min + waterRecycler.min
```

### 3.3 AICoordinator

AI 协同层不直接“发电”，它只做三件事：

- 读取状态
- 计算风险
- 改变负载分配

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `mode` | enum | `normal / warning / emergency / recovery` |
| `forecastWindow` | seconds | 推荐 `90` |
| `riskScore` | percent | 居住舱综合风险 |
| `lastAction` | string | 最近一次调度动作 |
| `explainText` | string | 面向用户的 AI 解释 |

## 4. Interface 模型

系统只保留三条接口，避免数值到处乱跑。

```text
SolarInput -> EnergyCabin.generation
EnergyCabin.availablePower -> HabitatCabin.loadAllocator
HabitatCabin.telemetry -> AICoordinator.riskModel -> HabitatCabin.loadAllocator
```

每个 tick 的顺序固定：

1. 更新外部输入：`solarInput`
2. 更新能源舱：`generation / batterySoc / batteryStress`
3. 计算居住舱需求：`normalDemand / criticalDemand / activeDemand`
4. AI 计算风险：`riskScore`
5. AI 输出调度：`loadCaps`
6. 居住舱执行负载分配
7. UI 渲染状态

## 5. Dependency 传播公式

### 5.1 可用功率

```text
rawDeficit = max(0, activeDemand - generation)
batteryBoost = batterySoc > reserveFloor ? min(rawDeficit, 24) : 0
availablePower = min(outputLimit, generation + batteryBoost)
```

解释：

- 发电够用时，不动用电池
- 发电不足时，电池最多补 `24 EU/s`
- 储能低于 `reserveFloor` 后停止非关键补偿，只保生命支持

### 5.2 储能变化

```text
netPower = generation - allocatedDemand
batterySocNext = clamp(batterySoc + netPower / 120, 0, 100)
```

解释：

- `120` 是演示缩放系数，用来让储能变化在几十秒内可见
- `allocatedDemand` 是实际分配后的负载，不是原始需求

### 5.3 电池压力

```text
deficitRatio = max(0, activeDemand - generation) / max(activeDemand, 1)
reserveRisk = max(0, reserveFloor - batterySoc) / reserveFloor
batteryStress = clamp(70 * deficitRatio + 30 * reserveRisk, 0, 100)
```

解释：

- 发电不足越多，电池压力越高
- 储能跌破应急线后，压力继续上升

### 5.4 生命支持指数

每个生命支持负载都有一个满足度：

```text
loadSatisfaction = allocatedPower / normalDemand
minimumSatisfaction = allocatedPower / minimumDemand
```

生命支持三项指数：

```text
oxygenIndex = clamp(oxygenAllocated / 18, 0, 1)
pressureIndex = clamp(pressureAllocated / 14, 0, 1)
thermalIndex = clamp(thermalAllocated / 22, 0, 1)
```

综合生命支持安全度：

```text
lifeSupportScore = 100 * (
  0.42 * oxygenIndex +
  0.36 * pressureIndex +
  0.22 * thermalIndex
)
```

解释：

- 氧气和舱压权重最高
- 温控可以降级，但长期不足会推高风险

### 5.5 居住舱风险

```text
criticalShortage = max(0, criticalDemand - availablePower) / criticalDemand
lifeRisk = 100 - lifeSupportScore
habitatRisk = clamp(
  45 * criticalShortage +
  35 * batteryStress / 100 +
  20 * lifeRisk / 100,
  0,
  100
)
```

这是主传播链的终点：

`Energy ↓ -> Battery Stress ↑ -> Life Support ↓ -> Habitat Risk ↑`

## 6. State 规则

### 6.1 模式阈值

| 模式 | 进入条件 | 退出条件 |
| --- | --- | --- |
| `normal` | `habitatRisk < 25` 且 `batteryStress < 35` | 风险升高 |
| `warning` | `habitatRisk >= 25` 或 `batteryStress >= 35` | 风险恢复或继续恶化 |
| `emergency` | `habitatRisk >= 55` 或 `availablePower < criticalDemand` | `habitatRisk < 40` 且 `availablePower >= criticalDemand` 持续 10 ticks |
| `recovery` | 从 `emergency` 回落且供能恢复 | `habitatRisk < 25` 持续 10 ticks |

### 6.2 状态滞后

所有模式切换必须有 `10 ticks` 稳定窗口。

原因：

- 避免 UI 在阈值附近抖动
- 避免 AI 频繁开关负载
- 让恢复过程看起来像系统在稳态回归

## 7. AI 调度规则

### 7.1 调度原则

AI 只做负载上限控制，不直接修改底层传感器数值。

```text
AI action = setLoadCap(loadId, capEU)
```

这样可以保持单一真相源：  
传感器负责读数，AI 负责策略，负载分配器负责执行。

### 7.2 Warning 模式

触发时：

```text
lighting.cap = 4
comfortLoad.cap = 6
publicDisplay.cap = 2
thermalControl.cap = 18
```

AI 文案：

> 太阳输入下降，预计能源舱将在 90 秒内进入高压区。已降低舒适性负载，保留生命支持冗余。

### 7.3 Emergency 模式

触发时：

```text
oxygenLoop.cap = 18
pressureControl.cap = 14
thermalControl.cap = 14
waterRecycler.cap = 6
lighting.cap = 2
comfortLoad.cap = 0
publicDisplay.cap = 0
```

AI 文案：

> 能源缺口已触及生命支持阈值。已关闭非关键负载，氧气循环、舱压维持与基础温控进入最高优先级。

### 7.4 Recovery 模式

恢复顺序：

```text
thermalControl: 14 -> 18 -> 22
waterRecycler: 6 -> 10
lighting: 2 -> 4 -> 8
comfortLoad: 0 -> 6 -> 12
publicDisplay: 0 -> 6
```

每 `5 ticks` 恢复一个台阶。  
如果 `habitatRisk` 重新超过 `40`，立即停止恢复并回到 `warning`。

AI 文案：

> 发电恢复稳定。系统将按生命支持、环境控制、生活负载的顺序逐步恢复，避免二次冲击。

## 8. 示例时间线

### 8.1 正常段

```text
t = 0
solarInput = 100
generation = 80
activeDemand = 90
batterySoc = 72
availablePower = 90
batteryStress = 8
lifeSupportScore = 100
habitatRisk = 6
mode = normal
```

说明：发电略低于需求，电池轻微补偿，系统稳定。

### 8.2 遮挡开始

```text
t = 12
solarInput = 62
generation = 49.6
activeDemand = 90
batterySoc = 68
availablePower = 73.6
batteryStress = 31
lifeSupportScore = 100
habitatRisk = 11
mode = normal -> warning
```

说明：生命支持暂时不受影响，但电池压力和能源缺口开始传播；系统因 `batteryStress` 接近阈值进入预警。

### 8.3 应急调度

```text
t = 24
solarInput = 42
generation = 33.6
activeDemand = 90
batterySoc = 55
availablePower = 57.6
batteryStress = 44
lifeSupportScore = 100
habitatRisk = 15
mode = warning
AI action = cut comfortLoad, cap lighting, cap thermalControl
```

执行调度后：

```text
activeDemand = 54
availablePower = 54
batteryStress = 26
lifeSupportScore = 91
habitatRisk = 11
mode = warning stable
```

说明：AI 没有让系统进入失控，而是通过裁剪非关键负载让风险回落。

### 8.4 深度遮挡

```text
t = 38
solarInput = 28
generation = 22.4
activeDemand = 54
batterySoc = 34
availablePower = 46.4
batteryStress = 41
lifeSupportScore = 84
habitatRisk = 19
mode = warning
```

如果继续下降：

```text
solarInput = 18
generation = 14.4
availablePower = 38.4
criticalDemand = 48
batteryStress = 51
lifeSupportScore = 76
habitatRisk = 32
mode = emergency
```

说明：一旦可用功率低于关键需求，即使综合风险尚未超过 `55`，系统也必须进入应急模式。

## 9. UI 映射

### 9.1 必须显示的主数值

| UI 字段 | 来源变量 | 推荐显示 |
| --- | --- | --- |
| 太阳输入 | `solarInput` | `%` |
| 发电功率 | `generation` | `EU/s` |
| 储能余量 | `batterySoc` | `%` |
| 电池压力 | `batteryStress` | `%` |
| 当前负载 | `activeDemand` | `EU/s` |
| 生命支持安全度 | `lifeSupportScore` | `%` |
| 居住舱风险 | `habitatRisk` | `%` |
| 系统模式 | `mode` | 文本 + 颜色 |

### 9.2 必须显示的因果链

UI 上需要有一条清晰链路：

```text
太阳输入 ↓
  发电功率 ↓
    电池压力 ↑
      AI 调度动作
        生命支持安全度稳定
```

这是整个 Demo 的灵魂。观众必须看见“变化如何传播”，而不是只看见几个数字同时跳动。

### 9.3 颜色建议

| 状态 | 颜色语义 |
| --- | --- |
| `normal` | 稳定白 / 低饱和蓝绿 |
| `warning` | 琥珀黄 |
| `emergency` | 红橙 |
| `recovery` | 青绿 |

颜色只表达模式，不承载额外业务含义。业务含义由数值和 AI 文案解释。

## 10. 验收标准

这份数值系统完成后，应满足以下验收：

1. 调低 `solarInput` 后，`generation` 必须同步下降
2. `generation` 低于 `activeDemand` 后，`batteryStress` 必须上升
3. `availablePower` 低于 `criticalDemand` 后，系统必须进入 `emergency`
4. AI 调度后，`activeDemand` 必须下降
5. 非关键负载下降时，`lifeSupportScore` 不应明显下降
6. 太阳输入恢复后，系统必须先进入 `recovery`，再回到 `normal`
7. UI 必须能解释至少一条完整传播链，而不是只展示最终状态

## 11. 非目标

本阶段不做以下事情：

- 不追求真实航天工程单位校准
- 不模拟完整热力学、气体循环或电池化学模型
- 不引入更多舱体争夺叙事中心
- 不让 AI 直接改写底层物理状态

当前重点是跑通 AI-Native SE 的最小闭环：  
`模型 -> 状态 -> 变化传播 -> AI 调度 -> 可解释反馈`
