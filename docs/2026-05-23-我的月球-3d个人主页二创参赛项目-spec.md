# 我的月球 3D 个人主页二创参赛项目 Spec

## 1. 项目定位

原项目是一个可交互探索的 3D 个人主页：用户操纵角色在低多边形广场中移动，靠近展板、影院、向导、社交雕塑等对象并触发弹窗内容。

本次二创不重写交互底座，而是把这套“探索式个人主页”改造成黑客松参赛项目：

`一个可探索、可触发、会通过数值变化自我调度的月球居住系统 Demo。`

换句话说，原项目的价值是“空间化浏览作品集”；二创后的价值是“空间化理解复杂系统如何运行”。

## 2. 二创原则

### 2.1 保留

- 保留 `3D 漫游`：用户仍通过键盘、鼠标或移动端摇杆在场景中移动。
- 保留 `靠近触发互动`：重要模块仍通过靠近、点击或按键打开信息面板。
- 保留 `低多边形 / 绘本式` 视觉方向：适合“我的月球”的亲近感，不把项目做成冷硬工程舱。
- 保留 `轻量静态部署`：优先复用当前构建产物的加载方式和静态资源目录。

### 2.2 替换

- 作品广场替换为月球居住社区核心场景。
- 披风小猫角色后续替换为月球居民或宇航小人。
- 作品展板替换为系统状态牌、舱体说明牌和调度日志牌。
- 马戏影院替换为主场景回放装置。
- 向导尼卡替换为 AI 协同层助手。
- 社交雕塑替换为系统子模块节点。

### 2.3 不做

- 不做完整月球基地沙盒。
- 不引入大量新舱体分散叙事。
- 不追求真实航天工程仿真。
- 不让用户在第一版里自由编辑所有参数。

当前目标是用最小交互闭环讲清楚：

`太阳输入下降 -> 能源舱输出下降 -> 电池压力上升 -> AI 调度负载 -> 生命支持稳定`

## 4. 场景布局

### 4.1 空间结构

第一版只需要一个主场景：

```text
月球居住核心区
├─ 中央：居住舱
├─ 一侧：能源舱
├─ 两舱之间：能量连接通道
├─ 前景：AI 任务控制台
├─ 周边：生命支持子系统节点
└─ 远端：场景回放舱 / 扩展舱占位
```

### 4.2 视觉层级

- 居住舱永远是视觉中心，因为它承载“人如何继续生活”。
- 能源舱必须有可见连接线或能量流，表达供给关系。
- AI 控制台需要靠近两舱中间，表达它是协同层，不是孤立聊天窗口。
- 子系统节点围绕居住舱分布，让氧气、舱压、温控、水循环成为可触发对象。

## 5. 舱体结构规则

结构补充的当前确认约束是：居住舱和能源舱不再使用传统圆柱舱、方盒舱或胶囊舱，而是采用“立体脊骨三角舱体”。两者外形语言统一，但接口角色不同：用户操控的居住舱是榫结构，能源舱是卯结构。

### 5.1 单舱形态

单个舱体是一个偏三角形的独立脊骨模块：

- 顶部保留脊骨感的立体曲面，提供未来感和识别度。
- 中间必须保留人能站立和通行的平面高度，不做纯雕塑造型。
- 三个方向都有内凹的大面，用作连接面和门洞区域。
- 底部采用三个支撑点，每个支撑点可理解为轮子或移动底盘单元。
- 舱体高度与轮子/底盘高度大约为 `1:1`，整体偏扁平但不能趴地。

设计判断：这个形态让“模块化拼接”成为视觉第一信号，而不是只在文案里解释模块化。

### 5.2 榫结构居住舱

用户操控模型是 `居住舱`，它承担移动、探索和主动对接职责。

居住舱接口规则：

- 居住舱是榫结构。
- 居住舱有三个凸起。
- 三个凸起都带连接口。
- 每个凸起都可以尝试对接外部凸起或凹入连接口。
- 用户移动的不是人形角色，而是居住舱本体。

设计含义：居住舱不是被动房间，而是一个可以自己寻找接口、拼接社区、接入能源的移动生活单元。

### 5.3 卯结构能源舱

能源舱承担固定能源节点和接入枢纽职责。

能源舱接口规则：

- 能源舱是卯结构。
- 能源舱有三个凹入接口。
- 居住舱接入能源舱凹入接口后，才能获得能源接入能力。
- 三个凹入接口都可以作为能源接入口。
- 能源舱不作为用户操控主体。

设计含义：能源舱不是普通背景建筑，而是系统能源网络的锚点。用户必须通过居住舱完成对接，才能让数值系统从“独立消耗”进入“能源协同”。

### 5.4 磁挫锁对接

当前对接方式采用 `frustrated magnetism`，中文表述为 `磁挫锁`。

对接规则：

- 同极之间形成稳定结构，而不是传统机械插销锁死。
- 居住舱凸起靠近能源舱凹入接口后，进入磁挫锁捕获范围。
- 捕获范围内先出现吸附/校准动画，再进入稳定锁定。
- 锁定后仍需手动确认能源接入，不自动并网。
- 能源接入成功后，数值系统才允许 `availablePower` 读取能源舱供给。

可视化要求：

- 对接前：接口发出弱脉冲，提示可连接。
- 捕获中：凸起与凹口之间出现短暂磁场线或干涉波纹。
- 稳定后：连接处出现稳定环形光，表示磁挫锁成立。
- 并网后：能量流从能源舱流向居住舱。

### 5.5 双门通道

每个舱体在连接面都有一扇门。两个舱体连接后，通道不是单门，而是双门：

- 舱体 A 有一扇门。
- 舱体 B 有一扇门。
- 需要打通时，两扇门同时内滑开启。
- 门的开启方式类似地铁屏蔽门，不外翻、不上翻。

这条规则很重要：它让“连接”和“通行”分开。磁挫锁成立只表示结构连接，双门开启才表示人员可以穿梭。

### 5.6 组合方式

居住舱和能源舱的榫卯角色区分后，组合规则改为：

| 模式 | 形态 | 用途 |
| --- | --- | --- |
| 居住舱接能源舱 | `居住榫 -> 能源卯` | 主场景能源接入 |
| 居住舱接居住舱 | `居住榫 <-> 居住榫` | 生活面积扩展、移动社区 |
| 多居住舱围绕能源舱 | `多个居住榫 -> 能源卯节点` | 社区核心、能源分配 |

### 5.7 旧方案废弃说明

`docs/结构补充.md` 中早期提到的“每个舱三面完全对称通用”和“底盘伸缩圆球榫卯”是历史草稿。当前 spec 以本节为准：

- 居住舱是榫结构。
- 能源舱是卯结构。
- 对接机理是磁挫锁。
- 旧的圆球伸缩机构不再作为主实现要求。

## 6. 核心交互流程

### 6.1 初始进入

用户进入场景后看到：

- 用户操控的居住舱已通过一个凸起与能源舱凹入接口稳定对接。
- 能量连接通道稳定发光。
- AI 控制台显示 `normal`。
- 系统状态牌显示初始数值。

初始关键数值来自数值 spec：

```text
solarInput = 100
generation = 80
activeDemand = 90
batterySoc = 72
availablePower = 90
batteryStress = 8
lifeSupportScore = 100
habitatRisk = 6
mode = normal
```

### 6.2 触发主事件

用户靠近或点击“场景回放舱”后，启动主事件：

`太阳遮挡开始`

系统进入自动时间线：

```text
normal -> warning -> emergency? -> recovery -> normal
```

视觉上必须同步发生：

- 月球环境光略微变暗。
- 能源舱输出通道变弱。
- 电池压力条上升。
- AI 控制台出现调度解释。
- 非关键模块逐步变暗。
- 氧气、舱压、基础温控保持稳定亮起。

### 6.3 用户探索子系统

用户可以靠近子系统节点查看解释：

- `氧气循环`：为什么优先级最高。
- `舱压维持`：为什么不可关闭。
- `基础温控`：为什么可以降级但不能关闭。
- `水循环`：为什么是关键支持但可短时降级。
- `舒适性负载`：为什么优先被切除。

每个节点不只是说明文字，还要显示当前分配功率：

```text
normalDemand / minDemand / allocatedPower / priority
```

## 7. 数值系统接入

### 7.1 单一数据源

所有 UI、面板、颜色、动画都必须读同一个模拟状态对象：

```ts
type MoonSystemState = {
  tick: number;
  mode: "normal" | "warning" | "emergency" | "recovery";
  energy: EnergyCabinState;
  habitat: HabitatCabinState;
  ai: AICoordinatorState;
};
```

### 7.2 场景驱动规则

3D 场景不能自己发明状态，只能映射数值 spec：

- `solarInput` 控制环境光、太阳能板亮度或阴影强度。
- `generation` 控制能源舱输出光带强度。
- `batteryStress` 控制能源舱告警色和压力仪表。
- `activeDemand` 控制居住舱总负载环。
- `lifeSupportScore` 控制生命支持节点稳定度。
- `habitatRisk` 控制整体告警层级。
- `mode` 控制全局 UI 状态色和 AI 文案。

### 7.3 AI 调度动作可视化

AI 调度不能只是一段文案，必须映射到对象状态：

| 调度动作 | 3D 表现 | UI 表现 |
| --- | --- | --- |
| `lighting.cap = 4` | 居住舱外部灯带降低亮度 | 照明负载从 `8` 降至 `4` |
| `comfortLoad.cap = 0` | 非关键生活模块熄灭 | 舒适负载标记为 `OFF` |
| `thermalControl.cap = 14` | 温控节点从全亮变成保底呼吸 | 温控显示 `reduced` |
| `oxygenLoop.cap = 18` | 氧气节点保持稳定亮起 | 氧气显示 `protected` |
| `pressureControl.cap = 14` | 舱压节点保持稳定亮起 | 舱压显示 `protected` |

## 8. 信息架构

### 8.1 常驻 HUD

常驻 HUD 只显示最关键的 5 个值：

- `mode`
- `solarInput`
- `batterySoc`
- `lifeSupportScore`
- `habitatRisk`

不要把所有数字摊在首屏，否则观众会看不出主因果链。

### 8.2 控制台面板

AI 控制台负责显示：

- 当前模式
- 触发原因
- 调度动作
- 预计恢复条件
- 一句话解释

### 8.3 子系统面板

子系统节点负责显示：

- 模块名称
- 正常需求
- 保底需求
- 当前分配
- 优先级
- 当前状态

## 9. 文案方向

整体表达应从“功能说明”转向“系统正在照顾一个家”。

推荐主标题：

`我的月球`

推荐副标题：

`在月球上，搭建一个会自己运转的家。`

AI 助手的语气要短、明确、可解释：

```text
太阳输入下降，能源舱进入高压区。
已降低舒适性负载，保留生命支持冗余。
氧气循环与舱压维持保持最高优先级。
```

## 10. 素材替换计划

当前阶段的素材策略先收敛为一句话：

`第一版默认优先复用仓库现有素材；只有在视觉语义明显冲突时，替换动作才延后到第二阶段。`

这意味着第一版不为了“看起来像月球”就强行引入新模型，而是先用现有模型、纹理、地标和程序化图形把交互与数值闭环跑通。

### 10.1 当前复用策略

- 当前主入口继续使用 `index.html + assets/my-moon-demo.css + assets/my-moon-demo.js`。
- 舱体、磁挫锁、节点和能量流优先用程序化图形 / 灰盒表达。
- 可复用的现有 GLB、纹理和环境图优先直接接入第一版；只有在语义明显冲突时，才延后到第二阶段替换。

## 11. 实现边界

### 11.1 第一阶段

- 保留当前 3D 漫游和互动触发框架。
- 替换文案、目标点、弹窗内容和 HUD 语义。
- 增加数值模拟状态机。
- 把数值映射到关键对象颜色、亮度和面板。
- 完成主事件回放。

### 11.2 第二阶段

- 替换居住舱、能源舱、角色和子系统节点模型。
- 增加磁挫锁捕获/稳定动画、内滑双门动画和手动并网确认。
- 增加更明确的月面地形。
- 增加能量流动画。
- 增加完整移动端适配检查。

### 11.3 第三阶段

- 增加可交互参数调节。

## 12. 验收标准

第一版二创完成后，应满足：

1. 观众进入 10 秒内能知道这是“我的月球”而不是原个人作品集。
2. 场景里只突出居住舱、能源舱和 AI 控制台，不被其他功能点分散。
3. 触发主事件后，数值按 spec 传播，而不是随机变化。
4. AI 调度动作能在 3D 场景中被看见，而不只是写在弹窗里。
5. 用户能通过子系统节点理解“为什么先保氧气、舱压、基础温控”。
6. 用户操控主体必须是居住舱本体，而不是默认人形角色。
7. 居住舱必须呈现三个凸起榫接口，能源舱必须呈现三个凹入卯接口。
8. 对接动画必须区分“磁挫锁稳定”和“双门内滑开启”两个动作。
9. 能源接入必须经过手动确认，并且确认后才影响数值系统。
10. 素材未最终替换时，灰盒也必须表达清楚职责和方向。
11. 本地静态预览可运行，移动端保留基本可操作性。

## 13. 好品味检查

- 可消除的特殊情况：不要为每个子系统写独立状态机，统一用负载表和优先级驱动。
- 超过三层缩进：数值计算、场景渲染、面板文案必须分层，不能塞进一个组件。
- 不必要抽象：第一版只需要一个主事件时间线，不需要通用任务编辑器。
- 单一真相源：所有对象视觉状态都从 `MoonSystemState` 派生。
- 结构不乱分支：居住舱只定义榫接口，能源舱只定义卯接口；连接稳定性统一交给磁挫锁规则。

## 14. 最小实现伪代码

```ts
function tickMoonSystem(state: MoonSystemState, event: MoonEvent): MoonSystemState {
  const energy = updateEnergy(state.energy, event);
  const demand = applyLoadCaps(state.habitat.loads, state.ai.loadCaps);
  const risk = calculateHabitatRisk(energy, demand);
  const ai = planAiAction(state.ai, energy, demand, risk);

  return {
    tick: state.tick + 1,
    mode: ai.mode,
    energy,
    habitat: applyAllocation(state.habitat, demand, energy.availablePower),
    ai,
  };
}
```

这段伪代码是实现边界：数值先动，场景只是结果投影。这样二创不会退化成换皮页面，而会真正变成一个可运行的系统 Demo。
